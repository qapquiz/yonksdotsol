# Data Model

Core data structures used throughout the app. Read this before modifying anything related to positions, tokens, or caching.

## PositionInfo (from `@meteora-ag/dlmm`)

The primary data structure returned by `DLMM.getAllLbPairPositionsByUser()`.

```
PositionInfo {
  publicKey: PublicKey              // The LB pair address (NOT a position address)
  tokenX: { mint: { address: PublicKey } }
  tokenY: { mint: { address: PublicKey } }
  lbPair: {
    activeId: BN                    // Current active bin ID
  }
  lbPairPositionsData: Array<{      // ⚠️ MULTIPLE positions per pair!
    publicKey: PublicKey             // Individual position address
    positionData: PositionData {
      lowerBinId: number
      upperBinId: number
      totalXAmount: BN
      totalYAmount: BN
      totalClaimedFeeXAmount: BN
      totalClaimedFeeYAmount: BN
      positionBinData: PositionBinData[]
      ...
    }
  }>
}
```

### Critical Relationships

| Concept                  | How to Calculate                                                           | Common Mistake                                 |
| ------------------------ | -------------------------------------------------------------------------- | ---------------------------------------------- |
| **Pair count**           | `positionsArray.length`                                                    | Don't use this as position count               |
| **Position count**       | `positionsArray.reduce((sum, p) => sum + p.lbPairPositionsData.length, 0)` | `positionsArray.length` only counts pairs      |
| **Token mints per pair** | `position.tokenX.mint.address` + `position.tokenY.mint.address`            | Mints are shared across positions on same pair |
| **In-range check**       | `calculateIsInRange(activeId, lowerBinId, upperBinId)`                     | Must check per lbPosition, not per pair        |
| **Position address**     | `lbPosition.publicKey.toBase58()`                                          | NOT `position.publicKey` (that's the pair)     |

### Key Insight: One Pair → Many Positions

A single trading pair (e.g., SOL/USDC) can have multiple open positions. Each position has its own bin range, amounts, and fees. The UI renders one `PositionCard` per `lbPairPositionsData` entry, not per `PositionInfo`.

```
Map<string, PositionInfo>         // keyed by pair address
  └── PositionInfo
        ├── tokenX, tokenY        // shared across all positions on this pair
        ├── lbPair.activeId       // shared — current price bin
        └── lbPairPositionsData[] // individual positions with different ranges
              ├── [0] { lowerBinId: 100, upperBinId: 120 }
              └── [1] { lowerBinId: 95,  upperBinId: 130 }
```

## TokenInfo (from `src/tokens/index.ts`)

```
TokenInfo {
  mint: string
  symbol: string
  supply: number
  decimals: number
  cdn_url: string                   // Token icon URL
  price_info: {
    price_per_token: number         // USD price
    currency: string                // "USD"
  }
}
```

- Fetched via Helius RPC `getAsset` method (not a standard Solana RPC call)
- Cached in `CacheManager` with 60s TTL via `getOrFetch`
- Batched at list level in `useBatchTokenData` — collects all unique mints, fetches once

## PositionUpnl (from `metcomet`)

```
PositionUpnl {
  positionAddress: string
  upnlWithFees: number              // PnL including unclaimed fees (SOL)
  upnlWithFeesPercent: number       // Percentage
  currentValueInSol: number
  initialDepositInSol: number
  unclaimedFeesInSol: number
}
```

- Fetched per-wallet via `getUpnlPerPosition()`
- Cached in `CacheManager` with 15min TTL
- Keyed by `positionAddress` (matches `lbPosition.publicKey.toBase58()`)
- **`upnlWithFees` already includes fees** — displaying both PnL and fees is intentional for transparency, but avoid implying they're additive

## Cache Keys

All generated by `src/utils/cache/cacheKeys.ts`:

| Helper                         | Pattern                     | Example                        |
| ------------------------------ | --------------------------- | ------------------------------ |
| `getUpnlPerPositionKey(addr)`  | `upnl_per_position:{addr}`  | `upnl_per_position:7xKX...3jP` |
| `getTokenDataKey(mint)`        | `token_data:{mint}`         | `token_data:So11...1111`       |
| `getOHLCVKey(pool, bucket)`    | `ohlcv:{pool}:{bucket}`     | `ohlcv:LBpair:2026-04-10T12`   |
| `getPythPriceKey(sym, bucket)` | `pyth_price:{sym}:{bucket}` | `pyth_price:SOL:2026-04-10T12` |

## Map vs Array Convention

- Positions stored as `Map<string, PositionInfo>` — keyed by pair public key
- UPNL stored as `Map<string, PositionUpnl>` — keyed by position public key
- Token data stored as `Map<string, TokenInfo>` — keyed by mint address
- Always use `.values()` / `.entries()` to iterate, `.get()` for lookup
